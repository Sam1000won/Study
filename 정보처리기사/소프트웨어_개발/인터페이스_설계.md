# 24.시스템 인터페이스 요구사항 분석
## 1. 시스템 인터페이스 요구사항 구성
- 시스템과 외부 시스템을 연동하는데 필요한 시스템 인터페이스 요구사항을 기술
## 2. 시스템 인터페이스 요구사항 분석
- 요구사항 분류하고 조직화하여 요구사항 명세를 구체화
1. 요구사항 분석은 소프트웨어 요구사항 분석 기법
2. 요구사항은 분해가 필요한 경우 적절한 수준
3. 분석시 누락된 요구사항이나 제한조건 추가
4. 상대적 중요도를 평가하여 우선순위
## 3. 시스템 인터페이스 요구사항 분석 절차
1. 요구사항 목록에서 시스템 인터페이스 관련 요구사항을 선별하여 별도로 시스템 인터페이스 요구사항 목록
2. 정의서, 현행 시스템의 대.내외 연계 시스템 현황 자료등 시스템 인터페이스 요구사항과 관련된 자료 준비
3. 명세서를 확인하여 기능적인 요구사항과 비기능적인 요구사항 분석
4. 요구사항 목록 및 기타 관련 자료들을 비교하여 요고사항을 분석하고 내용을 추가하거나 수정
5. 명세서, 인터페이스 요구사항 목록을 관련 이해관계자에게 전달
# 25. 인터페이스 요구사항 검증
## 1. 요구사항 검증
- 요구사항 명세서에 정확하고 완전하게 기술되어있는지 검토하고 개발 범위의 기준인 베이스라인 설정
- 오류가 발견되어 수정할 경우 비용이 많은 프로젝트일 경우 매우 중요
- 요구사항 검토 계획 수립 -> 검토 및 오류 수정 -> 베이스라인 설정
## 2. 인터페이스 요구사항 검토 계획 수립
- 검토 기준 및 방법 - 참여자 - 체크리스트 - 관련 자료 - 일정
## 3. 인터페이스 요구사항 검토 및 오류 수정
- 체크리스트 항목에 따라 인터페이스 요구사항 검토
- 목록과 시정 조치를 작성
- 검토 결과를 검토 관련자들에게 전달
- 시정 조치서를 작성할 경우 시정 조치 여부를 검토함
## 4. 인터페이스 베이스라인 설정
- 요구사항 명세서의 베이스라인을 설정
## 5. 요구사항 검증 방법
- 요구사항 검토(Requirements Review): 
    - Peer Review(동료검토): 작성자가 내용을 직접 설명하고 이해관계자와 결함을 발견하는 형태의 검토
    - Walk Through(워크스루): 검토회의 전에 미리 배포하여 사전 검토한 후 결점 발견
    - Inspection(인스펙션): 요구사항 명세서 작성자를 제외한 다른 전문가에게 검토
- Prototyping(프로토타이핑): 사용자의 요구사항을 정확히 파악하기 위해 견본을 만들어 최종 예측
- 테스트 설계: 요구사항은 테스트할 수 있도록 작성. 현실적인 테스트가 가능한지 검토
- CASE 도구 활용: 일관성 분석 요구사항 변경사항의 추적 및 분석, 관리, 표준 준수 여부 확인
## 6. 인터페이스 요구사항 검증의 주요 항목
- Completeness: 요구사항 누락 확인
- Consistency: 모순이나 충돌점 확인
- Unambiguity: 요구사항을 명확히 이해 할 수 있는가
- Functionality: How to -> What 중점
- Verifiability:  사용자의 요구를 만족, 사용자 요구 내용과 일치성 검토
- Traceability: 명세서와 설계서 추적 가능성
- Easily Changeable: 명세서의 변경을 명료하게 작성
# 26. 인터페이스 방법 명세화
- 송수신,오류 식별 및 처리 방안에 대한 문서로 명확히 정리
## 1. 시스템 연계 기술
- 시스템과 내 외부 시스템을 연계할 때 사용되는 기술
- DB Link: DB에서 DB Link 객체 이용 방식
- API/Open API: 송신 시스템의 DB에서 데이터를 읽어와 제공하는 app 인터페이스 프로그램
- 연계 솔루션: EAI 서버와 송-수신 시스템에 설치되어 클라이언트를 이용하는 방식
- Socket: 생성하여 포트를 할당하고 클라이언트의 통신 요청 시 클라이언트와 연결하여 통신하는 네트워크 기술
- Web Servive: WSDL과 UDDI, SOAP 연계
## 2.통신 유형
- 단방향성: 거래 요청만 하고 응답이 없는 방식
- 동기: 응답이 올 때까지 대기하는 방식
- 비동기: 시스템에서 거래를 요청하고 다른 작업을 수행하다 응답이 오면 처리하는 방식
## 3.인터페이스 처리 유형
- 데이터를 어떻게 처리할 것인가
- 실시간 방식: 처리해야 할 때 사용 방식
- 지연 처리 방식: 데이터 처리 비용이 많이 발생할 때 사용
- 배치 방식: 대량의 데이터를 처리 할때
## 4. 인터페이스 발생 주기
- 전송되어 인터페이스가 사용되는 주기를 의미
- 전송량을 고려하여 매일,수시, 주 1회 등으로 구분
## 5. 송,수신 방법 명세화
- 내,외부 인터페이스 목록에 있는 각각의 인터페이스에 대한 연계 방식, 통신 및 처리 유형, 발생 주기 등의 송수신 방법을 정의 명세
## 6. 송,수신 데이터 명세화
- 테이블 또는 파일 단위로 송,수신 데이터에 대한 명세 작성
## 7. 오류 식별 및 처리 방안 명세
- 오류를 식별하고 오류 처리 방안에 대한 명세를 작성하는 것
- ES10001: 연계 서버에 접속 불가
- ES50001: 데이터 변환 에러
- SD40001: 송신 시스템에서 데이터 조회 실패
# 27. 미들웨어 솔루션 명세
## 1. 미들웨어 개념
- 서버와 클라이언트 사이에서 원만한 통신이 이루어지도록 다양한 서비스를 제공
- 위치 투명성 제공
- 데이터 교환에 일관성을 보장
- 미들웨어의 내부 동작을 확인하려면 별도의 응용 소프트웨어 사용
- 1:1,1:N,N:M
## 2. DB
- DB와 클라이언트를 연결하기 위한 미들웨어
- 2Tier 아키텍쳐
## 3. RPC
- PC 호출하여 원격을 로컬처럼 다루는 것
## 4. MOM
- 메시지 기반의 비동기형 메시지를 전달하는 방식의 미들웨어
- 온라인 업무보다는 이기종 분산 데이터 시스템의 데이터 동기를 위해 많이 사용
## 5. TP-Monitor
- 트랜잭션 처리 모니터는 항공기나 철도 예약 업무에 처리 및 감시하는 미들웨어
- 사용자 수가 증가해도 빠른 속도 유지가능
## 6. ORB
- ORB 객체 요청 브로커는 객체 지향 미들웨어 코바 표준 스펙을 구현한 미들웨어
- 트랜잭션 처리와 모니터링 등을 추가로 구현
## 7. WAS
- 정적인 콘텐츠를 처리하는 웹 서버와 달리 사용자의 요구에 따라 변하는 동적인 콘텐츠를 처리하기 위한 상대
- 웹 환경을 구현하기 위한 미들웨어
- HTTP 세션 처리를 위한 웹 서버 기능뿐 아니라 미션- 크리티컬 한 기업 업무까지 java ejb 컴포넌트 기반 구형 가능
## 8. 미들웨어 솔루션 식별
- 개발 및 운영 환경에 사용될 미들웨어 솔루션을 확인하고 기록 작성
- 구매가 진행 중이거나 구매 예정인 소프트웨어  내역을 확인하여 개발 및 운영 환경에서 사용될 미들웨어 솔루션 식별
- 미들웨어 솔루션 목록을 작성
- 오류 및 누락을 확인하고 수정
## 9. 솔루션 명세서 작성
- 상세하게 기술하는 것
- 솔루션에 대한 제품안내서 및 설명 자료 등을 검토
- 사용 환경과 특징 등을 솔루션 설명 자료나 관람 담당자를 통해 검토
- 이해관계자 등에게 전달하여 오류 및 누락을 확인하고 수정
## 10. 미들웨어 솔루션 명세서 작성
- 솔루션 목록의 미들웨어 솔루션별로 관련 정보들을 상세하게 기술
- 목적, 솔루션에 대한 제품 안내
- 사용 환경과 특징을 설명,관련 자료 당당자 검토
- 계약사항이 존재하는 제품 안내서 및 기술 지원 담당자를 통해 검토
- 상세 정보, 제공 기능, 특징, 시스템 구성 환경 등에 대한 제약사항을 정리하여 솔루션 명세서 작성
