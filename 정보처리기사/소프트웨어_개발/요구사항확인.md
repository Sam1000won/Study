# 요구사항 확인
## 1. 소프트웨어 생명주기
- 개발 방법론의 바탕이 되는 것
- 정의하고 운영, 유지보수 과정을 각 단계로 나놈
- **IEEE의 소프트웨어 공학 표준 용어 사전** : 개발 운용 유지보수 폐기에 대한 처계적인 접근 방안
- **Fairley**: 비용과 기간 내에 소프트웨어를 체계적으로 생산하고 유지보수하는 데 관련된 기술적이고 관리적인 원리
- **Boehm**: 과학적인 지식을 소프트웨어 설계와 제작에 응용하는 것이며 개발,운영,유지보수하는 데 필요한 문서 작성 과정

### 기본 원칙
- 현대적인 프로그래밍 기술을 계속 적영
- 품질이 유지되도록 지속적 검증
- 관련 사항 및 결과에 명확한 기록을 유지

## 2. 폭포수 모델
- 가장 오래된 모델
- 두 개 이상의 과정이 병행해서 수행 ㄴㄴ
- 하향식 방법
- 검토 -> 계획 -> 분석 -> 설계 -> 구현(코딩) -> 시험 -> 유지보수

## 3. 프로토타입 모형
- 최종 결과물을 예측하는 모델
- 의뢰자, 개발자 모두 공동의 참조 모델
- 골격 코드
- 단기간 제작으로 비효율적인 언어나 알고리즘이 사용
- 요구 수집 -> 빠른 설계 -> 프로토타입 구축 -> 고객 평가 -> 프로토타입 조정 -> 구현

## 4. 나선형 모형
- 보헴이 제안/ 위험 분석 기능을 추가한 모델
- 점진적 모형
- 최소화하는 것을 목적
- 요구사항이 이해하기 어려운 경우 사용
- 계획 수립 -> 위험 분석 -> 개발 및 검증 -> 고객 평가
## 5. 애자일 모형
- 유연하게 대응할 수 있도록 주기를 반복하면서 개발과정을 진행
- 고객 소통을 맞춘 방법론
- Sprint, Iteration 짧은 개발 주기를 반복 결과물에 결과물에 대한 고객 평가 적극 수용
- 개발 작업을 진행
### 개발 모형
    - Scrum/ XP/ Kanban/ Lean/ Crystal/ ASD/ FDD/ DSDM/ DAD
### 애자일 선언
- 프로세스와 도구보다 개인과 상호작용 우선
- 문서보다 SW 가치
- 협상보다 협업
- 계획보다 변화에 반응에 가치를 둠
### 12가지 지침
1. 빠르고, 지속적인 고객을 만족
2. 요구사항 변경을 적극 수용
3. 개월이 아닌 주단위 SW 제공
4. 고객과 개발자가 함께 일함
5. 의지있는 팀, 개발 환경과 지원을 제공, 신뢰
6. 의견을 중요시
7. 진척도를 확인 하는 기준은 1차
8. 장려하고 일정한 속도로 계발
9. 지속적 관심 -> 민첩성 향상
10. 단순화
11. 주도적으로 조직적 팀으로 나옴
12. 정기적으로 고민하고 팀의 행동을 조정

## 2. 스크럼 기법
- 스스로 팀을 구성, 문재 해결
#### 제품 책임자(PO)
- 제품 이해도가 높고 책임지고 의사 결정
- 의견 종합과 요구사항 작성
- 백로그에 대한 우선순위 지정
- 책임자만이 백로그 작성/ 팀원은 추가만 가능
- 주기적으로 요구사항 갱신
#### 스크럼 마스터
- 조력자역할 통제 ㄴㄴ
- 문제점 공론화를 통해 같이 해결
#### 개발팀
- 책임자와 마스터를 제외한 팀원
- 7~8 명
### 3.프로세스
1. 제품 백로그
- 요구사항을 우선순위에 따라 나열 목록
- 지속적으로 업데이트
- 릴리즈 계획을 수립
2. 스프린트 계획 회의
- 단기 일정을 수립
- 요구사항을 개발자들이 나눠서 작업할수 있도록 지원(Task)
- 스프린트 백로그 : 작업 단위로 분할한 후 개발자별로 수행할 작업 목록
3. 스프린트
- 실제 개발하는 과정 2~4주간 진행
- 속도를 추정한 후 개발 담당에게 할당
- 직접 선별하여 담당할 수 있도록 함
- Task 할당은 / To_Do, In Progress, Done
4. 일일 스크럼 회의
- 15분간 짧은 회의
- 서서 진행, 남은 작업 시간은 소멸 차트 표시
- 스크럼 마스터는 장애 요소를 해결
5. 스프린트 검토 회의
- 부함한지 참석자 앞에서 테스트
- 한 주에 한 시간
- 피드백 후 백로그 업로드
6. 스프린트 회고
- 규칙 준수 여부, 개선점을 확인 후 기록
- 스프린이 끝난 시점에서 수행 혹은 일정 주기 실행

## 3. XP
- 고객 요구사항을 유연하게 대응하기 위해 고객 참여를 통해 생산성 극대화
- 요구사항을 빠르게 개발하는 것을 목표
- 릴리즈를 짧게 반복하고 요구사항 가시성 높임
- 소규모 인원 효율적
#### 핵심 가치
- 의사소통(Communication)/ 단순성(Simplicity)/ 용기(Courage)/ 존중(Respect)/ 피드백(Feedback)
### 1. XP 개발 프로세스
- 사용자 스토리(User Story)
    - 고객의 요구사항을 간소화 표현
    - 기능 단위로 구성, 간단한 테스트 사항 기제
- 릴리즈 계획 수립(Release Planning)
    - 부분적으로 완료된 제품을 제공하는 것
    - 부분 혹은 전체 완료 시점 일정 수립
- 스파이크(Spike)
    - 신뢰성을 높이고 위험은 감소 시키기 위한 프로그램
    - 처리할 문서 외의 다른 조건은 모두 무시
- 이터레이션(Iteration)
    - 하나의 릴리즈를 더 세분화 한 단위
    - 1~3주 정도의 기간
    - 새로운 스토리 작성 가능, 진행 혹은 차기에 포함 가능
- 승인 검사(Acceptance Test, 인수검사)
    - 하나의 이터레이션 안에 부분 완수된 것을 테스팅
    - 고객이 직접 수행
    - 오류 사항은 다음 이터레이션에 기제
    - 새로운 요구사항이 작성 혹은 상대적 우선상황 변경 가능
    - 다음 이터레이션 지행
- 소규모 릴리즈
    - 기능별 고객 반응 확인 요구사항에 좀더 유연하게 대응
    - 계획된 최종 테스트 수행, 최종 결과물을 고객에 전달
    - 다음 일정 진행
#### 주요 실천법
- Pair Programing
    - 개발 책임을 공동으로 나눠 가짐
- Collective Ownership
    - 권한과 책임을 공동으로 소유
- Test-Driven Development
    - 무엇을 하는지 명확히 알아야됨
    - 지속적 테스트를 위해 자동화
- Whole Team
    - 자신의 역할에 책임을 가지고 운영
- Continuosus Integration
    - 모듈 단위로 개발된 코드를 마무리 될때까지 지속 통합
- Design Improvement/ Refactoring
    - 기능의 변경 없이 단순화, 유연성 강화 등을 통해 재구성
- Small Releases
    - 짧은 릴리즈 기간을 짧게 반복함으로써 요구사항을 신속히 대응
## 4. 현행 시스템 파악
### 1. 현행 시스탬 파악 절차
1. 단계
- 구성
- 기능
- 인터페이스
2. 단계
- 아키텍쳐
- 소프트웨어 구성
3. 단계
- 하드웨어 구성
- 네트워크 구성
### 2. 시스템 구성 파악
- 주요 업무를 담당하는 기간과 이를 지원하는 지원 업무로 구분
### 3. 기능 파악
- 기능과 하부기능, 새부 기능을 구분하여 계층형 표시
### 4. 인터페이스 파악
- 데이터 형식, 통신규약, 연계 유형을 반드시 고려
### 5. 아키텍쳐
- 최상위 수준에서 계층별로 표현한 아키텍처 구성도 작성
- 가장 핵심이 되는 기간 업무 처리 시스템을 기준으로 표현
### 6. 소프트웨어 구성 파악
- 제품명, 용도, 라이센스 적용 방식, 라이선스 수 등 명시
### 7. 하드웨어 구성 파악
- 단위 업무 시스템들이 운영되는 서버 사양,수량, 이중화의 적용여부 명시
- 서버의 이중화 판단 기준: 서비스 기간 장애 대응 정책에 따라 결정
- 이중화가 적용되면 시스템 구축 난이도가 높아질 가능성도 고려
### 네트워크 구성 파악
- 서버 위치, 네트워크 연결 방식을 구성도로 작성
- 보안 취약성을 부석하고 적절히 대응 가능
- 장애 발생 원인을 찾아 복구 용도로 활용 가능

## 5. 개발 기술 환경 파악
### 1. 개발 기술 환경의 정의
- OS, DMS, MW 등 선정 할 사항을 기술, 오픈 소스 시 주의
### 2. OS
- 자원을 효율적 관리, 편리하고 효율적으로 사용할 수 있도록 제공
#### 고려사항
- 가용성/ 성능/ 기술 지원/ 주변 기기/ 구축 비용
### 3. DBMS
- 사용자와 데이터베이스 사이세서 요구에 따라 정보를 생산, 데이터베이스를 관리해주는 소프트웨어
- 구성,접근 방법, 유지관리 모든 책임을 짐
- 종속성과 중복성의 문제를 해결하기 위해 제안된 시스템 공용 관리
#### 고려사항
- 가용성/ 성능/ 기술 지원/ 상호 호환성/ 구축 비용
### 4. WAS(Wab Application Server)
- 사용자의 요구에 따라 변하는 동적인 컨텐츠를 해결하기 위한 `미들웨어`
- 접근, 세션관리, 트랜잭션 관리 등을 위한 라이브러리 제공
- 서버와 연동
#### 고려사항
- 가용성/ 성능/ 기술 지원/ 구축 비용
### 5. 오픈 소스 사용에 따른 고려
- 소스 코드를 공개적으로 라이센스를 만족하는 소프트웨어
- 종류, 사용자 수, 기술의 지속 가능성을 고려

## 6. 정의 사항
### 1. 요구사항의 개념 및 특징
- 서비스에 대한 설명과 운영되는 필요 제약조건
- 기준과 근거를 제공
- 이해관계자들간의 의사소통을 원활하게 도움
- 목표와 계획을 수립
### 2. 요구사항의 유형
#### Functional requirements
- 시스템의 목적을 사항
- 데이터를 저장하거나 연산을 수행해야 하는지에 대한 사항
- 시스템이 반듯이 수행해야 하는 사항
- 사용자가 시슽템을 통해 제공받기를 원하는 기능
#### Non-Functionalrequirements
- 하드웨어, 소프트웨어, 네트워크 시스템 장비구성에 대한 요구사항
- 속도,시간,처리량,동적,정적 적용량,가용성등 성능대 대한 요구사항
- 시스템,사용자에 대한 요구사항으로 다른 Sw,Hw,Interface, 정보 교환에 사용되는 프로토콜과의 연계도 포함하는 기술
- 자료 구축 변환을 위한 대산,방법,보안이 필요한 데이터를 구축하는 필요한 요구사항
- 성능 테스트, 구축된 시스템이 제대로 운영되는지를 테스트 요구사항
- 시스템의 데이터 및 기능,운영 접근을 통제하기 위한 요구사항
- 관리가 필요한 품질의 요구사항으로 가용성/ 정합성/ 상호 호환성/ 대응성/ 신뢰성/ 사용성/ 유지 관리성/ 이식성/ 확장성/ 보안성 등 구분 기술
#### Non-Functionalrequirements
- 설계,구축, 운영과 관련한 기술,표준,업무법-제도, 제약 조건
- 원한 수행을 위한 방법에 대한 요구사항
- 지원 사항이나 방안에 대한 요구사항
#### User requirements
- 사용자 관점에서 본 시스템이 제공해야 되는 사항
- 친숙한 표현으로 이해하기 쉽게 작성
#### System requirments
- 개발자 관점에서 본 전체가 사용자와 다른 시스템에 제공해야 할 요구사항
- 사용자 요구사항에 비해 전문적이고 기술적인 용어로 표현
- 소프트웨어 요구사항
### 3. 개발 프로세스
- 체계적으로 도출하고 분석 후 결과를 명세서에 정리 다음 확인 및 검증
- 도출 -> 분석 -> 명세 -> 확인
- 비지니스 목적 평가, 예산 정보 수집, 평가 보고서 판단이 선행
### 4.요구사항 도출
- 의견을 교환하여 요구사항을 식별하고 이해하는 과정
- 문제를 이해하는 것이 첫번째
- 이해관계자가 식별
- 의사소통이 중요함
- SDLC가 지속 반복
#### 주요기법
- 청취와 인터뷰
- 설문
- 브레인스토밍
- 워크샵
- 프로토타이핑
- 유스케이스
### 5. 요구사항 분석
- 불확실한 부분을 걸러내는 과정
- 비용, 일정에 제약 설정
- 상반되는 요구사항에 중재하는 과정
- 범위를 파악
- 적용하는 방법의 이해
- DFD, DD등이 사용
### 6. 요구사항 명세
- 모델을 작성하고 문서화 하는 것
- 필요한 것만 명확히 기술
- 이해하기 쉬우며 효과적으로 설계
- 잘못된 부분이 있으면 추적이 쉬워야함
- 구체적 명세를 위해 소단위 명세서 사용
### 7. 요구사항 확인(검증)
- 명세서가 정확한지 검증 및 확인 검토하는 과정
- 분석가가 요구사항을 정확히 이해 후 도출 했는지 필요
- 실제 방영 여부, 반발되는지 확인
- 오류 발생시 재작업 비용이 발생할 수 있을지 요구사항검증이 매우 중요
- 일관성,명료성, 기준 부합 여부, 누락 여부를 확인후 검증하는 것이 중요
- 이해관계자 검토 필요
- 모든 문서를 확인할 수 있는 것은 아님
- 일반적으로 도구를 사용하여 형성 관리 수행

## 7.요구사항 분석
### 1. 요구사항 분석 개요
- 실제 개발 첫 단계로 개발 대상에 대한 사용자 명세화 활동
- 타당성 조사 및 비용,일정 제약 설정
- 목표 산출, 해결법 결정
- 명확성, 일관성 분석 후 문서화
- 요구사항 분석 수행단계
#### 분석 방법
- 애자일/ UML/ DFD/ DD/ Mini Spec/ 개채 관개도/ 상태 전이도/ 제어 명세서 등 이용
### 2. 구조적 분석 기법
- 자료 흐름과 처리를 중심
- 도형 중심의 분석을 통해 사용자의 요구사항을 문서화
- 하향식 방법을 사용 시스템 세분화, 중복을 배제
- 품질 향상, 모든 단계 명세서 작성 가능
### DFD/ 버블 차트
- 자료의 흐름, 변환 가정 기능을 도형으로 기술 하는 방법
- 프로세스와 자료 저장소에 흐름과 처리를 중심으로 `구조적` 분석 기법
#### 구성
- Process
- Data Flow
- Data Store
- Terminator
#### 작성 지침
- Process를 거칠때 마다 새로운 이름 부여
- 출력 자료를 산출하기 위해서는 반드시 입력 자료가 발생해야 함
- 상위 단계 처리와 하위 자료 흐름은 서로 일치 해야됨
- 화살표가 반드시 출력 화살표가 아님
### 4. 자료 사전 (Data, MetaData)
- 자료 흐름도에 있는 자료를 상세의 정의하고 기록
- 자료흐름도 시각으로 자료에 대한 정보를 체계적이고 조직적으로 모아 사용
#### 기호
- {}: 자료의 반복
- (): 생략
- +: 자료 연결
- =: 자료 정의
- []/or: 자료 선택
- **: 주석
## 8. 요구사항 분석 CASE와 HIPO
### 1. CASE(자동화 도구)
- 요구사항을 자동을 분석하고, 명세서를 기술하도록 개발된 도구
#### 사용이점
- 비용 축소
- 변경이 주는 영향 추적 용이
- 결함,생략,불일치 등의 발견 용이
- 문서 표준화로 품질 개선
- 분석자들 간의 적절한 조정
#### 종류
- SADT
    - SoftTech 개발한 것으로 시스템 정의, 요구사항 분석, 설계를 위해 온 구조적 분석 및 설계 도구
- SREM = RSL/REVS
    - TRW사 실시간 처리 소프트웨어 시스템에서 요구사항을 명확히 기술하기 위해 개발
    - RSL: 요소, 속성,관계, 구조 구술이 필요
    - REVS: RSL에 요구되는 것을 자동으로 분석하여 분석명세서 작성
- RSL.PSA
    - 미시간 대학에서 개발
    - PSL: 문제 기술 언어
    - PSA: Psl기반으로 자동 분석하여 보고서 출력하는 문제 분석기
- TAGS
    - 시스템 공학 방법 응용에 대한 자동 접근 방법
    - 구성: IORL, IORL을 처리하기위한 도구, 기초적인 TAGS 방법론
    - IORL: 요구사항 명세 언어
### 2. HIPO
- 시스템의 분석 및 설계 문서화 할때 사용하는 기법
- 구성: 입,출력, 처리
- 하향식 소프트웨어 개발을 위한 문서화 도구
- 체계적인 문서 관리 가능
- 기호, 도표를 사용
- 기능과 자료의 의존 관계를 동시 표시
- 변경, 유지보수 용이
- HIPO Chart: 인터페이스계층 구조를 표현한 것
#### 차트 종류
- 가시적 도표(도식 목차): 시스템의 전체적 기능과 흐름을 보여주는 계층 구조도
- 총체적 도표(총괄도표, 개요 도표): 프로그램 기능을 기술한 것
- 세부적 도표(상세도표): 총체적 도표를 기본으로 상세히 기술
## 9. UML
### 1. UML개요
- 개발자,고객의 의사소통을 원할하게 해주도록 표준화된 객체지향 모델링 언어
- OMG가 표준
- 6개의 구조 다이어그램, 동작을 표현하는 7가지 행위 다이어그램이 있음
- 사물과 사물간의 관계를 용도에 맞게 표현
### 2. 사물
- 모델의 가장 중요한 기본 요소,
#### 종류
- 구조 사물(Stucural)
    - 시스템의 개념적,물리적 요소 표현
    - 클래스, 유스케이스, 컴포넌트, 노드 등
- 행동 사물(Behavioral)
    - 시간과 공간에 따른 요소들의 행위 표현
    - 상호작용, 상태 머신
- 그룹 사물(Grouping)
    - 요소들을 그룹으로 묶어 표현
    - 패키지
- 주해 사물
    - 부가적인 설명이나 제약을 표현
    - Note
### 관계
- 사물과 사물의 연관성을 표현
- 방향성은 화살표로 표시
- 서로 상관관계일 경우 실선으로 표시
- 객체수를 표현하는 다중도를 선위에 표시
#### 종류
- 1: 단일
- n: n개 객체
- 0..1: 1또는 0
- 0..* or *: 0또는 다수
- 1..*: 1 or 다수
- n..*: 적어도 n개 이상
- n..m: n개에서 최대 m개
##### 집합
##### 포합
##### 일반화
##### 의존
##### 실체화
### 4. 다이어 그램
- 사물과 관계를 도형으로 표현
- 가시화를 통해 의사소통의 도움
- 정적 모델링은 주로 구조다이어그램/ 동적 모델링은 행위 다이어그램
#### 구조 다이어그램 종류
- Class Diagram
    - 클래스와 클래스가 가지는 속성,클래스 사이의 관계 표현
    - 구조상의 문제점 도출 가능
- Object Diagram
    - 사물, 인스턴스의 특정 시점의 관계로 표현
    - 럼바우 객체지향 분석 기법에서 객체 모댈링에 활용
- Component Diagram
    - 실제 구현 모델인 컴포넌트 간의 관계나 컴포넌트 간의 인스턴스 관계 표현
    - `구현 단계에서 사용`
- Deployment Diagram
    - 결과물,프로세스,컴퓨넌트 등 `물리적 요소` 위치를 표현
    - 노드와 통신 경로 표시
    - `구현 단계에서 사용`
- Composite Structure Diagram
    - 복잡성을 가질경우 내부 구조를 표현
- Package Diagram
    - 구조화 한 페키지로 표현
#### 행위 다이어그램 종류
